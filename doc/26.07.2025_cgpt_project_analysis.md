AI‑Persona Framework – Repository Overview
Vision and Purpose
The AI‑Persona Framework aims to build a modular, local‑first framework for AI companions that preserve long‑term memory and allow users to define personalities for their agents. The README describes the vision as creating persistent AI assistants that remember details such as a user’s dog’s name, sports preferences and emotional experiences. The project wants to ensure that user data “resides locally, on your own server”
GitHub
. There is a secondary goal to enable the creation of “digital echos” of users – simulated personas that can provide feedback or advice based on historic knowledge
GitHub
. The framework is released under the GPLv3 license
GitHub

Core Data Flow
1 User Input and Routing
User messages can arrive through different interfaces (CLI, API, Discord). The router.py module is the entry point for message handling. It checks whether a message starts with a / indicating a tool command
GitHub
. If it is a command (e.g., /generate_image “cat with hat”), the router extracts the command and argument, looks up the corresponding tool in tool_registry.py, invokes it and returns the result to the user
GitHub
. If the tool is unavailable or fails, the router either sends a failure message back or appends a note for the LLM to handle
GitHub
. If the input is not a tool command, the router passes it to the engine (via engine.process_input) for normal dialogue processing
GitHub
.

2 Engine and Conversation Turn
The main controller (core/engine.py) orchestrates a conversation turn. Its run_conversation_turn function performs the following steps
GitHub
:

Session management – retrieve the user’s last chat session or create a new one using memory.chat_store.get_last_session / start_chat_session
GitHub
.

Context building – call memory.context_builder.build_context to embed the input and fetch relevant facts, vector matches and topics
GitHub
. The returned dictionary contains the embedded input and other context but the current engine uses only the embedded vector.

Emotion analysis – create an EmotionVectorGenerator and compute an emotion vector and tone for the user’s message
GitHub
. The system currently relies on a keyword‑based emotion model defined in shared/analysis/emotion_handler.py
GitHub
.

Logging user message – store the message, embedding and metadata to PostgreSQL via memory.chat_store.log_chat_message
GitHub
, then store the emotion vector via memory.emotion_store.store_emotion_vector
GitHub
.

Persona configuration – load the selected persona’s system prompt from a JSON configuration (defined by environment variable AGENT_CONFIG_PATH) using agents.loader.load_persona_config
GitHub
.

Assemble messages – fetch past conversation history from memory.chat_store.get_chat_messages and build a list of messages (system prompt, previous messages and the current user message)
GitHub
.

LLM call – send the assembled message list to the language model via core.llm_client.call_llm. The llm_client formats the messages using the llama.cpp <|im_start|> chat template and posts them to an LLM server configured via environment variables
GitHub
. The function returns the assistant’s reply and raw data
GitHub
.

Logging assistant response – store the assistant’s reply as a chat message in PostgreSQL
GitHub
.

The engine returns a dictionary containing the session ID, the user’s input, the assistant reply and the raw LLM output
GitHub
. A wrapper function (process_input) is provided for integration with the router
GitHub
.

3 Context Builder and Memory Retrieval
The context builder (memory/context_builder.py) is responsible for retrieving relevant context from the memory subsystems. Given a user_id and input_text, it performs these steps
GitHub
:

Embedding – embed the input text using the SentenceTransformer model (embedding.embed_text)
GitHub
.

Vector similarity search – query the Qdrant collection for the top‑k similar embeddings using memory.vector_store.search_similar_vectors
GitHub
 and package the payloads and scores.

Structured facts – fetch user‑specific facts from PostgreSQL via memory.fact_store.get_facts
GitHub
.

Topic relationships – extract topics from the vector hits and retrieve related topics from Neo4j via memory.topic_graph.get_related_topics
GitHub
.

It returns a dictionary with facts, vector results, related topics, raw input and the embedded vector
GitHub
. The current engine uses only the embedded vector, but this context will be needed for richer prompt construction in future versions.

Vector store (Qdrant) – provides functions to upsert embeddings and search for similar vectors

. The Qdrant collection is initialised with dynamic payload fields derived from emotion definitions
GitHub
.

Structured memory (facts) – functions to store, retrieve, update and delete user facts are defined in fact_store.py
GitHub
.

Graph memory (Neo4j) – topic_graph.py defines functions to create user–topic relationships with optional emotional metadata and fetch related topics

. The Neo4j initialisation script sets up constraints and a full‑text index for topics
GitHub
.

Emotion memory – emotion_store.py records emotion vectors and labels for each message and can retrieve the most recent emotion state for a user
GitHub
.

Embedding generation – embedding.py loads a SentenceTransformer model specified in the .env file and exposes functions to embed single texts or batches

Agent Management and Personalities
Personality information is stored either in JSON files (LLM-client/agents/personality_config.json) or in the user_personalities database table. The agents.loader.load_persona_config function reads a JSON configuration file (path provided via AGENT_CONFIG_PATH) and returns a dictionary containing fields such as name, description and system prompt
GitHub
. The agent_loader.load_user_agents function fetches all agents belonging to a user from PostgreSQL
GitHub
. A separate personality_loader.load_active_agent merges a user’s agent configuration with default emotions defined in shared/config/emotions.yaml
GitHub
.

Two command‑line scripts help manage agents. shared/scripts/agent_creator.py inserts or updates a personality record in the user_personalities table
GitHub
. shared/scripts/agent_creator_with_emotions.py performs the same operation and also seeds the emotional_relationships table with the agent’s default emotions
GitHub
. These scripts demonstrate how the framework intends to manage and update personas.

Interface and API
A simple command‑line interface (interface/cli_chat.py) demonstrates how a client might interact with the API: it repeatedly reads user input, sends it to a /chat endpoint on localhost:8000 and prints the response
GitHub
. The interface/api/app.py file uses FastAPI to expose a /chat endpoint. It loads the persona, recent history and emotion state (via a prompt_builder module that has not yet been implemented), builds a prompt, queries the LLM and stores the result
GitHub

Summary of Current Status
The repository is pre‑alpha. The documentation and README provide a detailed vision, but many modules are incomplete or placeholders. For example, the tool registry has only an image generator stub
GitHub
GitHub
, the context_builder.py referenced in the README does not exist in LLM-client, and some API modules reference missing functions. The memory layer code is more complete—scripts to create database schemas and modules to embed, store and retrieve memory exist, though they are not yet fully integrated into the engine’s prompt construction. The engine.py currently uses only the embedded vector and does not inject facts, topics or emotional history into the prompt.

Recommendations and Next Steps
Implement context‑aware prompts – The build_context function returns facts, vector hits and topics, but the engine currently ignores these. A prompt builder should assemble this information (including persona details and emotional state) into the messages sent to the LLM. You may take inspiration from the doc/flowchart.txt data flow: memory sources feed into a context builder, which then informs the LLM
GitHub
.

Agent selection at start‑up – To allow choosing an agent when a conversation starts, the engine could accept a personality_id parameter and persist it in the session. The router could provide a /switch_model command (as suggested in doc/dataflow.md) that updates the current session’s personality ID
GitHub
.

Tool command parsing – The router includes a simple regular expression to parse commands
GitHub
. To support arguments with spaces or additional parameters, consider a more robust parser (e.g., using shlex.split). Implement additional tools (web search, sandboxed code execution) and register them in tool_registry.py.

Complete missing modules – Several files referenced in the README (e.g., LLM-client/core/context_builder.py, core/prompt_builder.py) are missing. Creating these modules will tie together memory retrieval, persona configuration and LLM prompting.

Support echo simulation – The doc/echo.md outlines a plan for simulating users’ voices over time
GitHub
. To move toward this goal, implement scripts to export conversations and facts into a training corpus and design prompts that condition the LLM on a user’s style and preferences.

Script creation by the LLM – The idea of allowing the LLM to create its own scripts (e.g., via commands like /create_script and /run_script) is intriguing. To implement this safely, you could:

Add a tool in shared/tools that creates a file in a designated scripts directory when the LLM outputs /create_script filename.py <code>.

Add another tool to run scripts with restricted permissions (subprocess or exec in a sandbox). Integrate these tools into tool_registry.py and update the router to recognise the commands.

Ensure proper security measures to prevent arbitrary code execution. For example, restrict file paths, limit the available Python built‑ins and disable network access.

Missing file random_project_toughts.md – The prompt references a file named random_project_toughts.md; however, a search of the repository does not reveal such a file. If you have notes or ideas in another location, consider adding them to the doc/ directory so they can be reviewed and incorporated.

Conclusion
The AI‑Persona Framework repository provides a solid foundation for building a local‑first, memory‑based LLM assistant. The memory server modules are thoughtfully designed to capture semantic context, structured facts, topics and emotions, while the client orchestrates conversation flow and persona management. The project is still early in development; completing the missing modules, enhancing the prompt builder and expanding tool support will be important next steps. Careful integration of memory context and persona configuration will enable the framework to deliver the rich, emotionally aware AI companions envisioned in the documentation.
